# 設計レビューの観点

## Week 2でやること

要件定義書のOKが出たら、DB設計とAPI設計を行い、講師のレビューを受けてください。
**設計にGOが出るまで実装を始めてはいけません。**

理由：設計ミスを実装後に発見すると、コードの大幅な書き直しが必要になります。

---

## DB設計

### 作成するもの

1. **ER図**（draw.io / Figma / dbdiagram.io のいずれかで作成）
2. **DDL**（CREATE TABLE文）

### 提出方法

`submissions/[自分の名前]/ph5/er-diagram.png`（またはURL）
`submissions/[自分の名前]/ph5/schema.sql`

### 講師がレビューする観点

| 観点 | 確認ポイント |
|------|------------|
| 正規化 | 同じ情報が複数のテーブルに重複していないか |
| 主キー | 全テーブルにPKが設定されているか |
| 外部キー | リレーションが外部キーで表現されているか |
| 多対多 | 中間テーブルで表現されているか |
| NULL設計 | NOT NULL / NULL可能の判断が適切か |
| インデックス | 検索・JOINで使うカラムにインデックスがあるか |
| 命名 | テーブル名・カラム名がスネークケースで意味が明確か |

## API設計

### 作成するもの

OpenAPI形式（YAML or JSON）またはMarkdownの表形式でAPIを定義する。

### 最低限記載すること

```markdown
## API一覧

| メソッド | パス | 認証 | 説明 |
|--------|------|------|------|
| POST | /api/auth/login | 不要 | ログイン |
| GET | /api/tasks | 必要 | タスク一覧取得 |
| POST | /api/tasks | 必要 | タスク作成 |
...

## リクエスト・レスポンス例

### POST /api/tasks

**リクエスト**
```json
{
  "title": "タスクのタイトル",
  "status": "TODO"
}
```

**レスポンス（201 Created）**
```json
{
  "id": 1,
  "title": "タスクのタイトル",
  "status": "TODO",
  "createdAt": "2024-01-15T10:00:00"
}
```
```

### 講師がレビューする観点

| 観点 | 確認ポイント |
|------|------------|
| RESTful | HTTPメソッドとパスがREST原則に沿っているか |
| 認証 | 認証が必要なエンドポイントに認証が設定されているか |
| 命名 | パスがリソース名（名詞）になっているか |
| レスポンス | ステータスコードが適切か（200/201/400/401/404/500） |
| エラー | エラーレスポンスの形式が統一されているか |

---

## 設計レビューセッション（Week 2・30〜60分）

ER図とAPI仕様書が完成したら講師にレビューを依頼する。

### レビュー前に自分でチェックすること

- [ ] 要件定義書のMust機能が全てDBとAPIに反映されているか
- [ ] テーブル名・カラム名・APIパスの命名が一貫しているか
- [ ] 自分で設計を説明できるか（「なんとなくこうした」はNG）

### レビュー後

講師のFBをもとに設計を修正し、再度PRを更新する。
設計レビューでApprovedが出たらWeek 3（実装）開始。
