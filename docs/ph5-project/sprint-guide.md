# スプリントの進め方

## スプリントとは

**2週間という期間を区切って「この期間中に何を完成させるか」を宣言してから開発する手法**です。

現場でよく使われるアジャイル開発の基本です。計画→実装→振り返りのサイクルを回すことで、問題を早期発見できます。

## Week 3〜6の構成

| 期間 | スプリント | やること |
|------|-----------|---------|
| Week 3〜4 | Sprint 1 | Mustの機能（前半）を実装する |
| Week 5〜6 | Sprint 2 | Mustの機能（後半）+ テストを実装する |

---

## スプリントの開始前にやること（スプリント計画・1時間）

### 1. issueを整理する

GitHub Issuesに実装するタスクを全て起票する。

**issueの書き方：**

```
タイトル: [機能名] 具体的な実装内容
例: [タスク管理] タスク一覧取得APIを実装する

本文:
## やること
- TaskエンティティとRepositoryを作成する
- GET /api/tasks を実装する
- レスポンスにステータス・期限を含める

## 完了条件
- [ ] APIが動作する
- [ ] テストが書かれている
- [ ] PRが出せている
```

### 2. このスプリントでやるissueを決める

全issueを見て「2週間でできそうなもの」を選ぶ。
**詰め込みすぎは厳禁。** 余裕を持って3〜5issueが目安。

---

## スプリント中の1日の流れ

```
学習開始
  ↓
今日やるissueを1つ選ぶ
  ↓
featureブランチを切る
  git checkout -b feature/[名前]/issue-[番号]-[内容]
  例: feature/yamada/issue-12-task-list-api
  ↓
実装する
  ↓
テストを書く
  ↓
PRを出す（PRテンプレートを丁寧に記入）
  ↓
講師のレビューを待つ間に次のissueを始める
```

> **ポイント：** PRを出したら次のissueに進む。レビュー待ちで止まらない。

---

## スプリント終了時にやること（スプリントレビュー・30分）

### 1. デモ

動いているものを画面を見せながら説明する。

「〇〇機能を実装しました。このエンドポイントにリクエストを送ると…」

### 2. Sprint 振り返り（KPT）

| 項目 | 内容 |
|------|------|
| Keep | 良かったこと・続けること |
| Problem | 問題だったこと・困ったこと |
| Try | 次のスプリントで試すこと |

**記入例：**
- Keep：毎日PRを1つ出せた
- Problem：N+1問題に気づかず後から修正が必要だった
- Try：PRを出す前にSQLログを確認する習慣をつける

### 3. 次のスプリント計画

Sprint 2で何をやるかを決める。Sprint 1の振り返りを反映させる。

---

## issueとPRのリンク

PRの本文に `Closes #[issue番号]` と書くと、PRがマージされた時点でissueが自動的にCloseされます。

```markdown
## 何をしたか
タスク一覧取得APIを実装しました。

Closes #12
```

これにより進捗が自動的に可視化されます。
