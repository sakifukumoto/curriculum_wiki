# スプリントの進め方

## スプリントとは

**2週間という期間を区切って「この期間中に何を完成させるか」を宣言してから開発する手法**です。

重要なのは「完成しなかったら翌週に持ち越す」ではなく、**「完成しなかったらスコープを削る」**判断をすることです。これが現場と学習環境の最大の違いです。

---

## Week 3〜6の構成

| 期間 | スプリント | やること |
|------|-----------|---------|
| Week 3〜4 | Sprint 1 | Mustの機能（前半）を実装する |
| Week 5〜6 | Sprint 2 | Mustの機能（後半）+ テストを実装する |

---

## スプリント計画（各スプリント開始時・1時間）

### 1. Mustのissueを全て確認する

設計レビューでApprovedされた要件定義書を見て、全てのMust機能をissueに起票します。

**issueの書き方：**

```
タイトル: [機能名] 具体的な実装内容
例: [タスク管理] タスク一覧取得APIを実装する

本文:
## やること
- TaskエンティティとRepositoryを作成する
- GET /api/tasks を実装する

## 完了条件
- [ ] APIが動作する
- [ ] テストが書かれている
- [ ] PRが出せている

## 見積もり
（大・中・小のどれか。最初は感覚でOK）
```

### 2. このスプリントでやるissueを自分で決める

全issueを見て「2週間でできそうなもの」を選びます。

**現場と同じルール：**
- 詰め込みすぎは禁止
- 「絶対終わらせる量」より少なめに見積もること
- **スプリント終了時に終わらなかったissueは、翌スプリントに持ち越さない**

---

## スプリント中の進め方

```
featureブランチを切る
  git checkout -b feature/[名前]/issue-[番号]-[内容]
  ↓
実装する
  ↓
テストを書く（実装と同じブランチでOK）
  ↓
PRを出す（PRテンプレートを全項目丁寧に記入）
  ↓
講師のレビューを待つ間に次のissueに着手する
```

> **レビュー待ちで止まらないこと。** 次のissueに進む。

---

## スプリント終了時：必ずやること

### 1. 終わらなかったissueの「スコープ削減判断」

**このフェーズで最も重要な経験です。**

終わらなかったissueがある場合、**自分で**以下を決めてください。
講師が決めるのではなく、自分が決めることに意味があります。

| 選択肢 | 基準 |
|--------|------|
| A. スコープを削って今週中に終わらせる | 機能の核心部分だけ実装して残りはWon'tにする |
| B. 次のスプリントに持ち越す | 削れる部分がなく、この機能が他に影響する場合 |
| C. そもそもMustから外す | 「なくても動く」と判断できる場合 |

**判断したらPRまたはissueのコメントに理由を書く。**
「時間が足りなかった」は理由ではありません。
「〇〇の見積もりが甘かった。次回は△△を事前に確認してから見積もる。」が理由です。

### 2. デモ（10分）

動いているものをブラウザで見せながら説明します。

「〇〇機能を実装しました。このエンドポイントにリクエストを送ると…」

### 3. 振り返り（KPT・10分）

| 項目 | 書く内容の例 |
|------|------------|
| Keep | 毎日PRを1つ出せた |
| Problem | N+1問題に気づかず後から修正が必要だった |
| Try | PRを出す前にSQLログを確認する習慣をつける |

**Problemの深掘りルール：**
「時間が足りなかった」「難しかった」で終わりにしない。
「なぜ時間が足りなかったか」「何が難しかったか」まで掘り下げる。

### 4. 次のスプリント計画

Sprint 1の振り返りを必ず反映させること。
「同じProblemが2回出る」はKPTが機能していないサインです。

## 仕様が曖昧なときの対処

模擬プロジェクトでは意図的に仕様を曖昧にしている部分があります。

**「仕様が曖昧なときどうするか」も現場スキルです。**

やること：
1. 自分で仮決めして実装する
2. PRに「〇〇という解釈で実装しましたが、△△のケースはどう扱うべきですか？」と書く
3. 講師のFBを受けて修正 or そのままにする

やらないこと：
- 仕様が確定するまで実装を止める
- 全パターンに対応しようとしてオーバーエンジニアリングする

---

## issueとPRのリンク

```markdown
Closes #12
```

PRの本文にこれを書くと、マージ時にissueが自動Closeされます。

---

## 現場との対応関係

| このカリキュラムでの経験 | 現場での対応 |
|----------------------|------------|
| スプリント計画 | スプリントプランニング |
| 終わらなかったissueのスコープ削減 | バックログリファインメント・スコープ調整 |
| デモ | スプリントレビュー |
| KPT振り返り | レトロスペクティブ |
| 仕様が曖昧なときの対処 | 仕様確認・ドキュメント作成 |
