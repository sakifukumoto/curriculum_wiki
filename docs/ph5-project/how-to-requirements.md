# 要件定義のやり方

## 要件定義とは

「何を作るか」を言葉で明確にする作業です。

ここが曖昧なまま実装を始めると、**「こんな機能は聞いていない」「この仕様は変更したい」が実装後に発覚し、大量の手戻りが発生します。** 現場でよくある失敗パターンです。

## Week 1でやること

### Step 1：テーマ候補を読む（自分でやる・30分）

[テーマ候補](theme-candidates.md) を読み、作りたいテーマを1つ決める。
機能要件を読んで「これは実装できそうか？」をざっくり考えておく。

### Step 2：要件定義セッション（講師と一緒・60〜90分）

講師と1対1で以下を決めます。

#### セッションのアジェンダ

| 時間 | やること |
|------|---------|
| 15分 | テーマ確認・スコープの合意（何を作って何を作らないか） |
| 30分 | 機能一覧の洗い出し（付箋感覚でどんどん出す） |
| 20分 | 優先度付け（MoSCoW法） |
| 15分 | 疑問点・不明点の解消 |

#### MoSCoW法とは

| 優先度 | 意味 | 今回の方針 |
|--------|------|----------|
| Must | 絶対に必要 | 必ず実装する |
| Should | できれば欲しい | 時間があれば実装する |
| Could | あると良い | 余裕があれば |
| Won't | 今回はやらない | スコープ外と明示する |

> ⚠️ 未経験者は「全部Mustにしたい」という傾向があります。  
> 2ヶ月でできる量は限られています。Mustは5〜8機能程度に絞ること。

### Step 3：要件定義書を書く（自分でやる・3〜5時間）

セッションの内容をもとに、以下のフォーマットで要件定義書を作成してください。

```markdown
# 要件定義書

## プロジェクト概要
（何を作るか・誰のためか・何が解決されるかを3〜5行で）

## 対象ユーザー
（誰が使うか。例：社内の従業員、一般ユーザーなど）

## 機能一覧

### Must（必ず実装する）
- [ ] 機能名：説明
- [ ] 機能名：説明

### Should（時間があれば）
- [ ] 機能名：説明

### Won't（今回はやらない）
- 機能名：理由

## 画面一覧
（どんな画面が必要か。例：ログイン画面、タスク一覧画面など）

## 制約・前提条件
（技術的な制約、使用するフレームワーク、対象ブラウザなど）
```

### Step 4：要件定義書をPRで提出

`submissions/[自分の名前]/ph5/requirements.md` として提出。
講師がレビューしOKが出たら次のStep（DB設計・API設計）へ進む。

## OK基準

- [ ] Mustの機能が5〜8個に絞られている
- [ ] 「誰が何をする」が明確になっている
- [ ] Won'tに「やらない理由」が書かれている
- [ ] 画面一覧に必要な画面が全て挙がっている
