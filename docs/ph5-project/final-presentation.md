# 最終発表のフォーマット

## Week 8：最終発表（30〜45分）

このフェーズの締めくくりです。**「動くものを見せながら説明する」**ことが重要です。スライドだけの発表はNGです。

---

## 発表の構成

### 1. プロジェクト概要（3分）

- 何を作ったか
- なぜそのテーマを選んだか
- 使用した技術スタック

### 2. デモ（10分）

**実際に動いているアプリケーションを見せながら説明する。**

デモの流れの例（タスク管理アプリの場合）：
1. ユーザー登録・ログインを見せる
2. タスクを作成する
3. ステータスを変更する
4. ラベルで絞り込む

> ⚠️ デモは必ずリハーサルをしておくこと。本番で「動かない」は避けたい。

### 3. 技術的な見どころ（10分）

自分が工夫した点・難しかった点をコードを見せながら説明する。

説明すること：
- DB設計で悩んだ点と採用した理由
- 実装で一番詰まったこと・どう解決したか
- テストの設計方針

### 4. 振り返り（5分）

KPTの結果を共有する。

- Keep：良かったこと
- Problem：問題だったこと
- Try：次回やりたいこと

### 5. 質疑応答（5〜10分）

講師からの質問に答える。

**よく聞かれること：**
- 「なぜこのテーブル設計にしたのか」
- 「この処理をServiceに書いた理由は？」
- 「N+1問題はどこかで発生していないか」
- 「テストカバレッジはどのくらいか」

---

## 発表に向けて準備すること

**Week 7中に完了させること：**

- [ ] ローカルでアプリが動く状態にする（最低条件）
- [ ] デプロイ済みであれば本番URLを共有する
- [ ] README.mdに「起動方法」を書く
- [ ] KPTをまとめる

**発表前日にやること：**

- [ ] デモのリハーサル（1回は通しでやる）
- [ ] 「なぜこの設計にしたか」を3つ以上言えるようにする
- [ ] GitHubのPR履歴・コミット履歴を見返す

---

## 講師の最終FBの観点

| 観点 | 評価ポイント |
|------|------------|
| 設計力 | ER図・API設計が要件を満たしているか。設計の意図を説明できるか |
| 実装力 | レイヤー分離・例外処理・バリデーションが適切か |
| テスト | 主要な機能にテストが書かれているか |
| Git運用 | コミットの粒度・PR本文・レビューへの返信が適切か |
| 自走力 | 詰まったときの対処法・調べ方が身についているか |
| 説明力 | 設計の意図・実装の判断を自分の言葉で説明できるか |

> **最も重視するのは「説明力」です。**  
> 「なんとなく動いている」ではなく「なぜこう作ったか」が言えることが、現場で通用するエンジニアの証です。
