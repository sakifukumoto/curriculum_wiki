# 課題3-6：相互コードレビュー

## この課題の目的

**「レビューをする」経験を積む。**

レビューを受けるだけでなく、レビューをする経験は以下の理由で重要です。

- 「なぜこれが問題か」を説明するには、深い理解が必要
- 他の人のコードを読む力は、現場で毎日使うスキル
- 自分のコードの問題点を発見するセンスが養われる

---

## 進め方

### 1. ペアを決める

他のメンバー1人とペアを組みます。（講師が決めます）

### 2. 相手の課題3-3（SOLID・リファクタリング）のPRをレビューする

相手のPRを開いて、以下の観点でレビューコメントを書いてください。

**必ずコメントすること（全項目）：**
- [ ] SOLID原則のどれが改善されているか、具体的に説明する
- [ ] リファクタリング後のコードで、さらに改善できる点があれば提案する
- [ ] 「このコードを初めて読む人」として、わかりにくい点を指摘する

**コメントの書き方：**

```
✅ 良いコメント例
「StrategyパターンでNotificationを実装した判断は良いと思います。
 ただ、NotificationStrategyのインターフェースに
 sendの引数が String message だけなのは、
 将来的にメタデータ（送信先、優先度など）を渡したくなったとき
 インターフェースを変更する必要があります。
 最初からより汎用的な引数にする方法も検討してみてください。」

❌ 悪いコメント例
「LGTM！」
「良いと思います」
「ここはこう書いた方がいいです」（理由なし）
```

### 3. 相手からのレビューに返信する

相手からレビューコメントが来たら、以下の形式で返信してください。

```
[対応する場合]
ご指摘ありがとうございます。
〇〇という点は確かに問題だと思いました。
△△という理由で××に修正しました。

[対応しない場合]
ご指摘の点について検討しましたが、
今回は〇〇という理由で現状のままにしました。
この判断についてFBをいただけますか？
```

---

## レビューで使える観点チェックリスト

コードレビューで見るべき観点を優先順に示します。

### P1：必ず確認する
- [ ] バグがないか（論理エラー・NullPointerException・境界値）
- [ ] セキュリティ上の問題がないか（SQLインジェクション・認証の抜け）
- [ ] テストが書かれているか、十分か

### P2：重要
- [ ] 命名が意図を表しているか
- [ ] メソッドが1つのことだけをしているか（単一責任）
- [ ] 変更が来たときに影響範囲が小さいか

### P3：できれば
- [ ] 重複コードがないか
- [ ] パフォーマンス上の問題がないか
- [ ] コメントが必要な箇所にあるか

### やらなくていいこと
- インデント・スペースのスタイル指摘（フォーマッターの仕事）
- 「自分ならこう書く」という好みの押し付け

---

## OK基準

- [ ] 相手のPRに3つ以上のレビューコメントを書いた
- [ ] 全てのコメントに「なぜ問題か・どう改善するか」の理由が書かれている
- [ ] 相手からのコメントに全て返信した（対応・不対応の理由を明記）
- [ ] PR本文に「レビューをして気づいたこと・自分のコードに活かしたいこと」が書かれている

---

## 注意事項

- レビューコメントは批判ではなく提案として書く
- 「あなたのコードは悪い」ではなく「こうするとより良くなると思います」
- 受けるときも感情的にならず、技術的な議論として向き合う
- **レビューのやりとりは全て公開されます（他のメンバーも見られます）。現場と同じ状況です。**
