# 課題3-3：SOLID原則・リファクタリング

## この課題のゴール

SOLID原則を理解し、既存コードの問題点を指摘・修正できるようになる。

## 目安時間

15〜20時間

## SOLID原則とは

| 原則 | 意味 | 一言で |
|------|------|-------|
| S：単一責任の原則 | クラスは1つのことだけに責任を持つ | 1クラス1役割 |
| O：開放閉鎖の原則 | 拡張に対して開き、修正に対して閉じる | 既存コードを変えずに機能追加 |
| L：リスコフの置換原則 | 派生クラスは基底クラスと置き換えられる | 継承を正しく使う |
| I：インターフェース分離の原則 | 使わないメソッドへの依存を強制しない | 小さいinterfaceを使う |
| D：依存性逆転の原則 | 上位モジュールは下位モジュールに依存しない | interfaceに依存する |

## 仕様

以下のコード（意図的に問題を含む）をリファクタリングしてください。

```java
// 問題のあるコード
public class TaskManager {
    public void processTask(Task task, String action) {
        if (action.equals("notify_email")) {
            // メール通知のコード（50行）
        } else if (action.equals("notify_slack")) {
            // Slack通知のコード（50行）
        } else if (action.equals("save_db")) {
            // DB保存のコード（30行）
        } else if (action.equals("generate_pdf")) {
            // PDF生成のコード（40行）
        }
        // 将来的に通知方法が増えるたびにこのクラスを修正する必要がある
    }
}
```

### やること

1. 上記コードの問題点を **PR本文に列挙する**（どのSOLID原則に違反しているか）
2. SOLID原則に沿ってリファクタリングする
3. リファクタリング後に新しい通知方法（LINE通知）を**既存コードを変えずに**追加できることを示す

## OK基準

- [ ] 問題点の指摘が適切（少なくとも2つのSOLID原則違反を指摘）
- [ ] リファクタリング後に単一責任の原則が守られている
- [ ] 新しい通知方法を既存コードを変えずに追加できる構造になっている
- [ ] PR本文に「リファクタリング前後の比較と、なぜその設計にしたか」が書かれている
