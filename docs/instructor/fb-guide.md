# 講師用：FBガイド

## FBの目的

**「コードの正しさを確認する」ではなく「判断の根拠を問い続ける」**

コードが動いていてもFBを通す意味がないFBは多い。
このガイドに沿ってFBすることで、メンバーに「判断力」を育てる。

---

## FBの基本姿勢

### OKのときも「なぜOKか」を伝える

多くの講師はNGのときだけ理由を伝える。
しかしそれだと「間違えないようにする」思考が育つ。

**OKのときこそ理由を言語化させる。**

```
❌ 悪いFB例
「LGTM」（これだけ）

✅ 良いFB例
「Serviceに例外処理を集約した判断は良いですね。
 なぜControllerではなくServiceに書いたか、
 自分の言葉で説明してみてください。」
```

### 答えを渡さない。問いを渡す

```
❌ 悪いFB例
「N+1問題が発生しています。
 @EntityGraph(attributePaths = {"tasks"})を追加してください。」

✅ 良いFB例
「SQLログを確認してみてください。
 何回クエリが発行されていますか？
 それはなぜだと思いますか？」
```

### 「なぜAではなくBか」を必ず掘り下げる

PRテンプレートの「捨てた選択肢」欄が空or薄い場合は必ず返す。

```
「他の実装方法としてXという方法もあります。
 なぜそれを選ばなかったか考えてみてください。」
```

## フェーズ別のFB重点

### Ph.0〜Ph.1（基礎期）

**重点：「エラーを自分で読もうとしているか」**

- エラーメッセージを読んだか確認する
- 「何を調べたか」を必ず聞く
- この時期はヒントを多めに出してOK。詰まりすぎて意欲が落ちる方が問題

**FBで聞くこと**
- 「このエラーを見たとき最初に何をしましたか？」
- 「なぜこのメソッドをstaticにしましたか？」

### Ph.2（Spring Boot）

**重点：「レイヤーの役割を理解しているか」**

- Controller/Service/Repositoryに書くべきでないコードが紛れ込んでいないか確認
- N+1問題はSQLログを見て確認する（コードを見るだけでは不十分）

**FBで聞くこと**
- 「このロジックをServiceに書いた理由は？Controllerに書いたらどうなりますか？」
- 「SQLログを見せてください。何回クエリが発行されていますか？」
- 「このエンドポイントに不正なリクエストが来たときどうなりますか？」

### Ph.3（設計・品質）

**重点：「トレードオフを意識した判断ができているか」**

- 「なぜこのDB設計にしたか」を必ず口頭or文章で説明させる
- 設計に正解はない。「なぜこう判断したか」の根拠を問う

**FBで聞くこと**
- 「このテーブル設計の代替案は何かありますか？なぜそれを選ばなかったのですか？」
- 「このテストが落ちたとき、何が壊れたとわかりますか？」
- 「このメソッドが500行になったとき、どこから手をつけますか？」

### Ph.4（インフラ）

**重点：「エラーログから原因を特定できるか」**

- ログを読んで原因を特定する力を確認する
- 「AWSで何ができるか」より「なぜこのサービスを使うか」を問う

### Ph.5（模擬PJ）

**重点：「判断の説明責任を持てているか」**

- 設計のレビューでは「なぜこうしたか」を必ず言語化させる
- スコープ削減の判断を自分でさせる（講師が決めない）
- KPTの「Problem」欄が「時間が足りなかった」だけになっていたら掘り下げる

**FBで聞くこと**
- 「この設計で1年後に機能追加が来たらどこを変えますか？」
- 「このコードを引き継いだ人が最初に疑問に思うことは何だと思いますか？」

## 差し戻し時のテンプレート

```
## [課題名] 差し戻し

### 確認してほしいこと
（コードの具体的な箇所と問い）

### ヒント
（答えは書かない。調べ方・考え方のヒントだけ）

### 参考
（あれば）

修正後はコメントに「修正しました。〇〇という理由で△△に変更しました。」と返信してください。
「修正しました」だけはNGです。
```

## FB時間を効率化するために

週1〜2時間の制約のなかで全員にFBするには、以下を意識する。

**1. PRテンプレートの「捨てた選択肢」欄を読む**
ここが薄ければ、コードを読む前に差し戻す。これだけで時間を大幅節約できる。

**2. 優先度P1だけ必ず見る、P3は時間があれば**
（優先度ルールは[FB基準](../ph5-project/design-review-checklist.md)参照）

**3. デバッグ課題はログとStep 2の回答を先に確認**
「どこで詰まったか」が事前にわかるので、FBの的が絞れる。
