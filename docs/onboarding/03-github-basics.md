# GitHubの使い方

このカリキュラムでは、全ての課題提出をGitHubを通じて行います。  
ここではその基本操作を説明します。

## Gitとは・GitHubとは

| 用語 | 説明 |
|------|------|
| **Git** | コードの変更履歴を記録するツール（PC上で動く） |
| **GitHub** | Gitのデータをインターネット上で管理・共有できるサービス |

Gitは「日記帳」、GitHubは「日記帳を保管するクラウドストレージ」のようなイメージです。

---

## 基本的なGit操作

### よく使うコマンド

```bash
# 現在の状態を確認する
git status

# 変更したファイルを「次のコミットに含める」と宣言する
git add ファイル名
git add .  # 全ての変更を追加

# 変更を記録する（日記に書く）
git commit -m "コミットメッセージ"

# GitHubに送る
git push origin ブランチ名

# GitHubから最新を取得する
git pull origin ブランチ名

# ブランチを作る
git checkout -b 新しいブランチ名

# ブランチを切り替える
git checkout ブランチ名

# ブランチ一覧を確認する
git branch
```

## コミットメッセージのルール

このカリキュラムでは、コミットメッセージに**プレフィックス（接頭辞）**をつけます。

```
プレフィックス: メッセージ（日本語OK）

例：
feat: タスク一覧取得APIを実装した
fix: 404エラーのレスポンスが返っていなかったのを修正
test: TaskServiceのテストを追加
docs: READMEに起動手順を追記
refactor: ServiceクラスにDB処理を移動
```

| プレフィックス | 使う場面 |
|--------------|---------|
| `feat:` | 新しい機能・実装の追加 |
| `fix:` | バグ修正・レビュー指摘の修正 |
| `test:` | テストコードの追加・修正 |
| `docs:` | README・コメントの更新 |
| `refactor:` | 動作を変えないコード整理 |

## ブランチのルール

### ブランチとは

**ブランチ**は「作業用のコピー」です。  
メインのコードを壊さずに作業するために使います。

### このカリキュラムのブランチ命名規則

```
feature/[自分の名前]/[フェーズ]-[課題番号]-[内容]

例：
feature/yamada/ph0-task-0-3-git-basics
feature/yamada/ph2-task-2-2-rest-api
```

**禁止事項：mainブランチに直接pushしない**

mainブランチは「完成したコードだけ置く場所」です。  
必ずfeatureブランチを切って作業し、PRを通じてマージします。

## PRの出し方

PR（Pull Request）は「このコードをレビューして、OKならマージしてください」という依頼です。

### 手順

1. featureブランチにpushする

```bash
git push origin feature/yamada/ph0-task-0-3
```

2. GitHubで `curriculum_wiki` を開く
3. 「Compare & pull request」ボタンをクリック
4. PRテンプレートに沿って本文を記入する
5. 「Create pull request」をクリック

### PRテンプレートの書き方

PRを作成すると、本文に自動でテンプレートが表示されます。  
**3つの項目を全て記入してからPRを出すこと。**

```markdown
## 何をしたか
（実装した内容を具体的に。1〜3行）

## なぜこう実装したか（設計の意図）
（採用した理由を自分の言葉で。「調べたらこう書いてあった」はNG）

## 迷った点・わからなかった点
（正直に書く。「ない」は疑わしい。ここに質問を書いてもOK）
```

## レビューへの返し方

講師がPRにコメントをつけたら、以下の手順で対応します。

### NG（修正が必要）の場合

1. コメントを読んで修正する
2. 同じブランチにpushする（PRは自動で更新される）
3. コメントに返信する

**返信の書き方：**
```
✅ 修正しました。
〇〇という理由で△△に変更しました。
```

「修正しました」だけはNGです。**なぜそう修正したか**を必ず書いてください。

### OK（Approve）の場合

講師がApproveしたらマージします。  
（Ph.3まではマージは講師が行います）

## 困ったときのGitコマンド

```bash
# 直前のコミットを取り消す（ファイルの変更は残る）
git reset --soft HEAD~1

# 変更を全て元に戻す（注意：変更が消える）
git checkout .

# コンフリクトが起きたファイルを確認する
git status

# ブランチのログを確認する
git log --oneline
```

> ⚠️ `git reset --hard` は変更が消えるので慎重に使うこと。  
> 迷ったら講師に聞いてください。
